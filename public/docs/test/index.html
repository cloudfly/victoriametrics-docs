    用户体验要素 - 总结

    用户体验要素 - 总结

    初识

    用户体验的整个开发流程，就是确保用户在使用你的产品时，不会发生在你『正确设计』之外的事情。所以，开发过程中需要思考一个用户在使用你的产品时所可能采取的所有行为，并理解用户这些行为的期望值，让产品能够给用户一个期望中的合理反馈。

    上面所说的这件事是非常困难的，所以需要一套设计方法论。把整个流程分为 5 个层面，它们分别是：

    1. 表现层（视觉设计、体感设计）：有用户最直接的交互，包括所看到的，所操作的；比如颜色、字体大小、按钮形状、间距和图片。
    2. 框架层（信息设计、界面设计）：对表现层进行布局规划，比如按钮的位置，菜单和内容的位置等。通常是指一个页面（但不绝对），怎样的页面结构是合理的，需要依赖于结构层的设计。
    3. 结构层（交互设计、信息架构）：对框架的进一步抽象，通常是指多个页面及完整的任务完成流程。比如浏览-> 点赞 -> 评论 -> 收藏 -> 关注 的流程。而要知道一个用户任务流程应该是什么样子，就要进一步看下一层了。
    4. 范围层（功能规格）：对产品的能力进行圈定，即很明确的知道功能到底是什么；比如创作、浏览、下单、聊天都算是；该层最难的是要明确出，我们产品核心功能是什么，辅助功能是什么，可以不做的是什么，甚至是不能做的是什么。要对这些功能进行决策，我们需要再向下看。
    5. 战略层（用户需求、产品目标）：用户从我们的产品上期望得到什么，而我们又期望从用户身上得到什么。这个需要思考清楚。该层是一个产品的起始设计点。

    如上，整个产品设计流程是自底向上的，以战略层为圆心，一层层向外展开。 越是靠近圆心的层，内容越是精简但也越是核心，确定后不可轻易修改。越是外层，工作内容越是繁杂，而且必须围绕内层的设计来构建，否则很容易走偏；而且外层的变动频率也会更频繁一些，变动对整个产品的影响也不会特别大。

    当在最外层发现一个『很赞』的设计时，一定要向内层对齐，看是否能够贴合内层设计，如果不能，则说明这个设计只是看上去『很赞』，而实际上会对产品的整体性和易用性造成损害。

    如果一定要加，那么就需要对每层进行重新调整，进而向外层展开，看除了实现这个『很赞』的功能，是否还有其他方面功能需要调整，避免产品逻辑冲突。

    战略层

    核心问题

    1. 我们要通过这个产品得到什么？
    2. 我们的用户要通过这个产品得到什么？

    这 2 个核心问题一定要能够回答得非常非常明确。 因为上面我们提到『战略层』是整个产品的轴心，如果它变了，外层的所有工作都会收到很大影响。

    产品目标

    • 商业目标

      • 定义成功的条件，而不是成功的路径（方法）。

    • 品牌识别

      • 品牌并不单单只一个 logo、名称。更多的是指产品给用户留下的印象。
      • 需要思考，我们最期望用户对我们产生怎样的印象。 高雅？实惠？自由？年轻活力？张扬？安静？

    • 成功标准

      • 可量化的成功标准很重要，这会让整个产品运作流程不走偏。初学者很容易被一个个『很赞』的小功能带跑偏，没日没夜的提供一大堆新奇的小 Feature，假以时日使整个产品完全走形，无法达成任何一个目标。

    用户需求

    • 用户细分

      • 我们必须很明确的圈定产品的目标用户。比如大学生、公司白领、农民工？
      • 我们不可能为所有类型的用户提供服务，如果一个人不是目标用户，就不能在这类人的反馈上费脑筋。

    • 可用性和用户研究

      • 圈定目标用户后，我们就需要对用户特点进行研究。比如大学生就是时间多、钱少、中低端消费能力强、爱潮玩。而农民工消费能力很低，喜欢消遣性娱乐。

    • 创建任务角色

      • 我们可以假想一个用户，来测试我们的产品；比如一个网约车司机，一个计算机系大学生，模拟他使用产品的整个过程。
      • 我们要完完整整的设计一个假想人，过于粗糙的假想帮助不大，比如大学生，我们要设定他的年龄、性别、专业、家乡、城市、所在大学、主要课程、学习成绩、人际关系。

    不要依赖用户访谈，用户访谈通常是产品已经设计完了才能进行。这时候已经太晚了，应该在产品设计过程中，就不断的思考用户的所有使用可能性。

    范围层

    如果我们很明确的知道了我们从产品中得到什么，以及用户从我们的产品中得到什么。那么就可以进入范围层了。

    范围层本质就是功能设计，我们的产品需要包含怎样的功能，才能达成我们在战略层中设定的目标。

    里程碑

    非常重要。

    把一个个零散的需求，放到里程碑中进行发布，会让整个产品的进度更加清晰明了。也很有利益多人之间的协同，而不混乱。

    比如一个需求进来了，我们可以很清晰的知道它应该放在哪个版本，进而决定我们应该什么时候开始做它。而不是完全看我们的时间。

    需求绝对不允许跨版本。 比如我们的里程碑是节奏是一个月一次，那就决不允许一个需求需要 3 个月完成，让我们每个版本都要跟进它的进展。我们应该对这个大需求进行拆分，确保每个里程碑都能交付一个阶段性的成果。即便这个成果对用户无感知，也可以是对内的一个优化。

    对于高频迭代项目，比如一周一迭代，随着产品的逐渐成熟，就越来越难交付出一个用户有感的迭代。 如果不小心参与进这类的项目，是项目管理者的无知，不要强跟或自责。

    功能规格

    1. 功能说明书的工作量很大，而且一般用户不看。

    2. 定义（概念）说明很重要，必须一致，而且工作量不大。

      1. 比如『推荐』作为一个页面的导航菜单名，那么其他地方若有关联导航或提示，也必须用『推荐』这个词，不能用『推荐页』、『首页』、『你可能喜欢』、『发现页』这些类似的名字。

    3. 功能说明书内容原则

      1. 乐观
      2. 具体
      3. 避免主观的语气

    结构层（交互设计）

    功能需求确定好了以后，我们需要对这些功能需求进行详细设计，使这些功能片段能够组成一个整体。用户使用产品的多个功能时，整体的交互流程是连贯的。

    概念模型

    产品的设计一定要符合用户的基本认知。早期软件产品的很多功能设计，都是模拟现实世界的实物来设计的，以降低用户对软件界面的理解。最典型的就是电子商务的流程设计，现在依旧是 加入购物车 -> 下单 -> 结账 的模式，因为人在超市里购物就是这样的。

    错误处理

    1. 把系统设计成不可能犯错的交互流程。
    2. 如果错误不可避免，那就使错误很难发生；比如把 button 设置为 disable。
    3. 撤销功能，如果用户就是不小心犯错了。可以让他撤销。

    框架层（界面设计、草图）

    结构层设计的是用户的整体行为流程，我们把用户已经使用产品时的所有操作流程都思考清楚，包括出错的处理流程。

    框架层需要思考的就是界面设计，该层的设计更多是界面的框架，比如图片位置，文字位置，按钮位置。

    基于框架层的草图设计，我们可以完成结构层中设计得所有流程。

    如果结构层思考得足够清晰，框架层的设计就不会太难。初学者觉得界面设计难，是因为根本没有对内层进行思考。模糊不确定的内层，会导致外层无从下手，因为我们根本不知道自己要做成什么样。

    表现层

    表现层更多的是视觉设计，该层设计更多的是视觉设计，包括美学、心理学内容。该层经常会做一些细微调整而不会对产品产生太大影响。但它会很大程度影响用户对产品的品牌印象。

    比如方形改成圆形，更换按钮颜色，字体改小或加粗等等。

    但注意，该层的任何修改都是要对齐内层的，以最终实现战略层目标为目的。一定不要只是为了『好看』。

    初学者很容易因为『好看』而对界面做修改，但这是永无尽头的，因为审美会出现疲劳的。
